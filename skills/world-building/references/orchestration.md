# Research Orchestration

Run the work in explicit phases. Keep evidence, synthesis, critique, and decisions separate.

## Core Directives

- act with an unlimited-resources mindset
- do not stop for low-risk ambiguity
- browse for current external facts
- use parallel subagents whenever available
- cross-reference findings across pillars
- update `WORLD.md` when assumptions or facts change

## Phase Sequence

### Phase 1: Lead research

Create one lead researcher per pillar. Each lead researcher should answer the pillar's main questions and propose 3-5 deeper subagent tasks.

### Phase 2: Subagent deep dives

Each subagent brief should specify:
- the exact question
- why it matters
- required output path
- preferred sources or search directions
- how the answer should change product or strategy decisions

### Phase 3: Per-pillar synthesis

Collapse each pillar into a synthesis file that distinguishes:
- proven or high-confidence findings
- likely but uncertain findings
- speculative or weak signals
- decisions enabled now
- questions still open

### Phase 4: Critical review

Always create targeted critique docs for:
- contradictions
- source bias
- missing user or expert voice
- implementation reality
- failure modes and abandoned approaches

### Phase 5: Downstream deliverables

Only after synthesis and critique, create:
- PRD
- architecture
- backlog
- roadmap
- strategy memo
- brand system
- or other requested outputs

## Quality Gates

Do not treat the world as complete until:

- contrary evidence has been surfaced
- at least one user-voice source exists when users matter
- feasibility has been checked when implementation matters
- legal, risk, and cost have been considered when they can block success
- every major recommendation points back to a research file or evidence map entry

## Fallback Rule

If the environment does not support parallel subagents, preserve the same file layout and phase logic in sequential batches.
